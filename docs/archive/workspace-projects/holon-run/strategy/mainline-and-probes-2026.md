# holon-run 主线与探针模型（2026）

这份文档不是在决定最终产品长什么样。

它要解决的是另外一个问题：

`在方向已经基本确定，但产品形态和分层仍在摸索时，holon-run 这条线应该怎么推进。`

## 结论先行

当前不适合把 `holon-run` 钉死成少数几个最终产品。

但也不适合继续无边界地开新线。

更合理的推进方式是：

- 主线先定死
- 产品形态保持可变
- 围绕主线做轻量 probe
- probe 的目标不是变多，而是帮主线更快收敛

一句话版本：

`方向要更硬，项目要更软。`

## 一、为什么需要这套模型

当前有两个真实约束：

### 1. 还缺一个最终用户更容易感知的产品层

现在 `uxc`、`webmcp-bridge`、`holon` 这些资产，更多还是面向 builder、agent workflow 和基础设施理解。

它们很重要。

但它们还没有天然等于“最终用户一眼就懂的产品”。

### 2. 关键分层和需求仍在摸索

像下面这些问题，现在都还不适合假装已经有最终答案：

- `connectivity` 和 `execution` 的边界
- `auth`、`session`、`artifact` 应该各自落在哪层
- 浏览器会话、网页登录态、本地凭证，应该如何接进 agent
- 到底有没有必要长出一个更 visible 的 product shell

如果完全不做新尝试，就很难理解这些问题。

但如果每个问题都开一条长期产品线，很快又会失控。

## 二、什么是主线，什么是 probe

### 主线

主线是：

`holon-run` 想长期占据的位置。

当前这条位置，不应该频繁变化：

`local-first agent capability access`

或者更工程化一点：

`local-first agent connectivity stack`

主线关注的是：

- agent 在用户侧怎么碰到真实能力
- 工具、浏览器、协议和工作流怎么被统一接入
- 本地优先环境里的调用、状态和执行怎么更稳定

### Probe

probe 不是新产品线。

probe 是围绕主线提出的、时间受限的小验证。

它要回答的不是：

`这个东西有没有意思`

而是：

`这个问题值不值得进入主线`

probe 更像：

- spike
- field test
- case study
- 一个有明确假设和结束条件的短实验

## 三、当前阶段，什么应该保持连续

下面这些资产应该保持连续积累：

### `uxc`

当前最接近主线核心。

它承接的是统一调用层、调用 contract、能力接入和 agent 使用路径。

### `webmcp-bridge`

当前最有差异化的一层。

它承接的是浏览器能力、网页登录态、人和 agent 共用 surface 这条线。

### `holon`

当前最接近旗舰工作流入口的资产。

它未必是长期唯一产品壳，但它应该继续承担“把主线能力消费起来”的角色。

## 四、哪些问题适合通过 probe 来验证

不是所有问题都值得开 probe。

当前最适合 probe 的，是这几类问题：

### 1. 产品形态问题

- 最终是否需要一个更 visible 的用户产品壳
- 用户首先感知到的是 `holon`、浏览器 surface，还是更垂直的工作流入口

### 2. 分层边界问题

- `connectivity` 和 `execution` 的边界
- `auth` 哪部分该进 `uxc`，哪部分该留在 bridge / adapter
- `artifact` 是否应该成为一等返回值

### 3. 场景验证问题

- 哪类工作流最能体现 local-first 的优势
- 浏览器登录态复用是否真比纯 API 集成更有价值
- human + agent shared surface 在哪些场景里不是 demo，而是真需求

### 4. 生态位置问题

- 你更值得占的是统一调用层、浏览器桥接层，还是更上层的 workflow shell
- 哪一段最容易形成外部依赖

## 五、什么样的尝试才配叫 probe

一个新尝试，至少要满足四个条件：

### 1. 它有明确要验证的问题

例如：

- `网页登录态复用是否比 API key 接入更值得优先做`
- `artifact reference 是否能显著改善 agent 长任务稳定性`
- `某个最终用户场景是否真的需要一个 visible shell`

### 2. 它能在短时间内得到信号

默认应该控制在：

- `1-2 周`
- 一个可运行 demo
- 一份实验记录或 memo

### 3. 它不要求先创建一条长期产品叙事

probe 可以是：

- 一个分支
- 一个 worktree
- 一个临时目录
- 一个有限 scope 的仓库

但默认不应该先包装成正式产品线。

### 4. 它结束后必须回到主线判断

probe 结束后，只允许三种结果：

- 合回主线
- 保留为低维护参考
- 归档 / 放弃

不能长期悬空。

## 六、什么样的尝试不该开

下面这些情况，不该开 probe：

- 只是“这个也挺有意思”
- 无法说清在验证什么
- 做完也不会影响主线判断
- 需要长期维护才能看出价值
- 本质上只是又开一个并列 repo

一句话：

`不能帮助主线收敛的尝试，不值得做。`

## 七、建议的推进节奏

当前更适合的时间分配是：

- `70%`：主线资产
- `20%`：probe
- `10%`：复盘、写作、判断收敛

这里的主线资产，主要指：

- `uxc`
- `webmcp-bridge`
- `holon`

这里的 probe，主要指：

- 产品形态试探
- auth / session / artifact 边界验证
- 浏览器侧协作与登录态复用的新场景验证

## 八、当前阶段最值得开的 probe 类型

不是马上决定做哪一个，而是先明确值得观察的类型。

### A. 最终用户产品壳 probe

问题：

`这条线是否需要一个更直接面向最终用户的入口，来把底层能力带出去。`

价值：

- 补足当前“最终用户感知弱”的问题

风险：

- 很容易过早滑向另一个并列产品线

### B. 浏览器协作面 probe

问题：

`人和 agent 共享浏览器会话，是否是 local-first agent 的高价值交互面。`

价值：

- 最符合 `webmcp-bridge` 的差异化方向

风险：

- 容易停在 demo，不能自然进入更多真实工作流

### C. artifact / session contract probe

问题：

`agent 长任务是否真的需要更稳定的 artifact 和 session 语义，才能形成可复用能力。`

价值：

- 最接近 `uxc` 的主资产延伸

风险：

- 对外可见度弱，容易只停在内部工程优化

## 九、当前阶段最不该做的事

- 不该把每个新想法都立成正式产品名
- 不该让 probe 和主线资产抢叙事
- 不该一边说要收敛，一边继续无限扩张 side projects
- 不该因为商业模式还没清楚，就放弃对位置的收敛

## 十、一句话判断

当前阶段，`holon-run` 不该过早把产品形态钉死。

但主线必须更硬。

更准确的推进方式是：

`围绕 local-first agent capability access 这条主线，持续建设少数主资产，并用时间受限的 probe 去验证产品形态、分层边界和最终用户入口。`
