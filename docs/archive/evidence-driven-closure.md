# Evidence-Driven Closure And Completion

Date: 2026-04-18

Related issue:

- `holon-run/holon#195` `run_once should not stop at analysis or claim completion without landed work`

Related RFCs:

- [docs/result-closure-contract.md](result-closure-contract.md)
- [docs/objective-delta-and-acceptance-boundary.md](objective-delta-and-acceptance-boundary.md)
- [docs/continuation-trigger-contract.md](continuation-trigger-contract.md)
- [docs/runtime-spec.md](runtime-spec.md)

This RFC proposes a runtime refactor for deciding when work is actually
complete.

The core goal is simple:

- stop treating assistant prose as the main completion signal
- move completion and waiting decisions onto explicit runtime evidence

## Problem

Today, `Holon` still mixes three different questions:

1. has the current turn settled?
2. has the current objective actually been satisfied?
3. what text should be shown to the operator?

That creates a fragile path to `completed`:

- the runtime sees no active waiting condition
- the turn produces a terminal brief
- the final delivery prompt says the task appears complete
- closure falls through to `completed`

This is fragile for implementation-heavy work.

In particular:

- the assistant can stop after analysis and still produce a plausible terminal
  summary
- a polished final brief can hide the fact that no code landed
- `Sleep` reasons and final text become de facto completion signals even though
  they are only text
- `run_once` can converge on a terminal-looking answer rather than a satisfied
  acceptance boundary

This is exactly the failure mode surfaced by `#195`.

## Goal

This RFC answers four questions:

1. when is a turn execution pass settled?
2. when is the active objective semantically complete?
3. what evidence is required for completion?
4. how should operator-facing delivery relate to closure?

## Non-Goals

This RFC does not define:

- final prompt wording
- the exact verification tool UX
- a full automatic semantic parser for operator intent
- the final delegated-task merge format
- every future acceptance policy variant

Those can build on this model, but should not define it.

## Core Judgments

### 1. Completion must not be inferred from assistant prose

The runtime should not decide `completed` from:

- "done"
- "completed"
- "I fixed it"
- "the task appears complete"
- a polished final summary

Those may be useful delivery text, but they are not sufficient evidence.

### 2. Turn settlement and objective completion are different layers

A turn can settle without the objective being complete.

Examples:

- the model finished an analysis pass but has not implemented the fix
- the model is blocked on operator clarification
- the model delegated blocking work and must wait for a result

So `Holon` should separate:

- execution settlement
- semantic closure

### 3. Implementation objectives require positive completion evidence

For implementation-like work, `completed` should require positive evidence such
as:

- landed file mutation
- delegated result with changed output
- verification success when required
- explicit grounded no-op when allowed

Absence of waiting is not enough.

### 4. Operator delivery is not the closure source of truth

The operator still needs a clear final summary.

But that summary should be derived from validated runtime evidence and accepted
closure state. It should not decide closure by itself.

### 5. `Sleep` remains posture, not completion

`Sleep` can still mean:

- no immediate work remains in the current pass
- the runtime may suspend until a future trigger

But `Sleep` should not itself mean:

- the objective is complete

### 6. No-op completion must be explicit and grounded

Sometimes the correct result is "no code change is needed."

That should still be allowed, but only when:

- the current objective allows no-op completion
- the agent provides an explicit grounded rationale
- the runtime accepts that rationale as sufficient evidence

## Model

This RFC introduces four new pieces:

- `turn_settlement`
- `objective_contract`
- `evidence_ledger`
- `closure_claim`

## 1. Turn Settlement

`TurnSettled` answers one question only:

- has the current execution pass finished its internal work?

This is close to Codex's runtime notion of a completed turn.

Recommended conditions:

- the current model stream has reached terminal provider status
- all tool futures started by the turn have drained
- there is no pending immediate continuation generated by that same turn
- there is no unresolved runtime-side prompt for the current turn

`TurnSettled` should not answer:

- whether the overall objective is done
- whether the final operator text is good enough

## 2. Objective Contract

The runtime should hold an explicit contract for what kind of completion is
required.

Minimum shape:

```rust
enum ObjectiveIntent {
    Implement,
    Analyze,
    Review,
    Answer,
}

struct ObjectiveContract {
    intent: ObjectiveIntent,
    require_landed_change: bool,
    require_verification: bool,
    allow_grounded_noop: bool,
}
```

Recommended defaults:

- issue-fix or coding `run_once` work should default to `Implement`
- `Implement` should usually require landed change
- `Implement` should usually require verification
- grounded no-op should be opt-in, not the default

The key point is that completion expectations become runtime-visible rather than
prompt-only.

## 3. Evidence Ledger

The runtime should accumulate closure evidence as structured facts.

Minimum first version:

```rust
enum CompletionEvidence {
    FileMutation { files: Vec<String> },
    VerificationPassed { command: String },
    VerificationFailed { command: String, exit_status: i32 },
    DelegatedResult { task_id: String, changed_files: Vec<String> },
    NoOpGrounding { rationale: String, basis: Vec<String> },
    ExternalBlocker { reason: WaitingReason },
    OperatorQuestionPending,
}
```

These records should be produced by runtime reducers and tool reducers, not by
post-hoc text scraping.

Likely sources:

- `Write` / `Edit` / `ApplyPatch` -> `FileMutation`
- worktree or child-task rejoin -> `DelegatedResult`
- verification tool success or failure -> verification evidence
- explicit wait primitive -> blocker evidence
- explicit no-op close claim -> `NoOpGrounding`

This ledger should be inspectable and auditable.

## 4. Closure Claim

The agent should not directly set `ClosureOutcome`, but it may propose a typed
closure claim.

Recommended primitive:

```rust
CloseObjective {
  outcome: "completed" | "waiting" | "failed",
  mode: "implemented" | "noop" | "analysis_only" | "blocked",
  summary: String,
  evidence_refs: Vec<String>,
}
```

This is not trusted automatically.

The runtime should validate it against:

- the objective contract
- the evidence ledger
- active blockers
- runtime errors

## Decision Algorithm

At the end of a settled turn, the runtime should evaluate closure in this
order.

### 1. Runtime failure beats everything

If the runtime sees unrecovered runtime failure, the closure decision is:

- `failed`

### 2. Explicit blockers become `waiting`

If the runtime sees explicit blocking evidence, the closure decision is:

- `waiting`

Examples:

- operator input is required
- delegated blocking work is still running
- external trigger is required
- timer-based resumption is required

### 3. `completed` requires validated evidence

If the agent claims completion, the runtime should validate the claim.

For `Implement` objectives:

- `implemented` requires landed change evidence
- if verification is required, it also requires successful verification
- `noop` requires `allow_grounded_noop = true` plus explicit no-op grounding
- `analysis_only` should not satisfy the contract

For `Analyze` or `Review` objectives:

- `analysis_only` may satisfy the contract if no stronger completion evidence
  is required

### 4. Missing evidence should not silently fall through to `completed`

If there is no validated completion path, the runtime should not treat the
objective as complete just because no waiting reason is currently visible.

Instead, it should do one of:

- continue execution
- transition to `waiting`
- transition to `failed`

depending on the blocker or failure type.

## Layer Separation

This RFC recommends three explicit layers.

### Layer 1: `TurnSettled`

Answers:

- did this execution pass fully drain?

### Layer 2: `ClosureDecision`

Answers:

- is the current objective completed, failed, or waiting?

This must depend on:

- objective contract
- evidence ledger
- blocker state
- runtime error state

### Layer 3: `OperatorDelivery`

Answers:

- what should the operator read?

This may use:

- validated closure claim summary
- evidence ledger
- tool history
- verification records

But it should not be the main closure source of truth.

## Implications For Current Holon Logic

This RFC implies several changes to the current architecture.

### 1. `completed` must stop being the default fallthrough

The current closure logic effectively says:

- if there is no waiting reason and no runtime error, treat the unit as
  completed

That rule should be removed for implementation-style objectives.

### 2. `run_once` should stop depending on terminal brief text

`run_once` should not require a terminal-looking brief before it can conclude.

Instead it should wait for:

- a settled turn
- a validated closure decision

### 3. Terminal delivery should become a report step, not a completion oracle

Prompts like:

- "The task appears complete"

should not be used to decide whether the work is actually complete.

If terminal delivery remains, it should happen after closure has already been
decided by the runtime.

### 4. `Sleep` should no longer carry completion semantics

`Sleep` text can still help explain the state, but closure should not depend on
its content.

## Migration Plan

The lowest-risk migration order is:

1. introduce `ObjectiveContract`
2. introduce `EvidenceLedger`
3. change closure evaluation to require positive completion evidence for
   implementation objectives
4. add `CloseObjective`
5. remove `run_once` dependence on terminal brief presence
6. demote terminal delivery and `Sleep` text to reporting-only surfaces

This keeps runtime behavior understandable while gradually removing fragile
text-based closure paths.

## Testing Implications

This RFC implies new regression coverage.

At minimum:

- analysis-only output must not complete an implementation objective
- a `completed` claim with zero landed change must be rejected
- verification-required implementation work must not complete without
  verification evidence
- grounded no-op completion must only succeed when explicitly allowed
- waiting and failure states must not depend on terminal brief text

## Relationship To Existing RFCs

This RFC does not replace the existing closure model.

Instead it tightens it:

- [docs/result-closure-contract.md](result-closure-contract.md) should be
  updated so `completed` requires positive evidence rather than merely the
  absence of waiting
- [docs/objective-delta-and-acceptance-boundary.md](objective-delta-and-acceptance-boundary.md)
  should be updated so acceptance boundaries can reference observable evidence
- [docs/runtime-spec.md](runtime-spec.md) should distinguish `TurnSettled`,
  `ClosureDecision`, and `OperatorDelivery`

## Summary

The key design change is:

- `Holon` should stop asking "did the assistant sound finished?"
- and start asking "did the runtime observe evidence that satisfies the active
  contract?"

That is the simplest way to make completion robust for headless coding and
delegated runtime workflows.
